# Phase 12: Dual-Database Retail Infrastructure Execution Plan

## User Review Required

> [!WARNING]
> This plan will fully replace the existing `DummyJSON` data source with `data-generator-retail`.
> It also unifies PostgreSQL and Neo4j into the same Docker cluster and expands the ETL pipeline to process Relational Schema (PostgreSQL) -> Graph Schema (Neo4j).

> [!CAUTION]
> **Data Quality Pre-Condition (Lesson Learned)**
> The `data-generator-retail` package produces only **130 unique products** duplicated ~800x to reach 104,000 rows, with `faker.lorem.paragraph()` as descriptions (meaningless Latin filler).
> This renders Knowledge Graph extraction useless because zero real entities (brands, materials, scenarios) exist in the text.
> **Fix**: The `diversify_data.py` Python Combinatorial Injection Engine MUST run dynamically to guarantee 104k perfectly unique dataset combinations locally before Postgres ingestion.

### Pipeline Refactoring (Neo4j ETL)

#### [MODIFY] `backend/ecommerce-graphrag/ingestion/async_fetch.py`

**Dual-Model Architecture:**

- **Part A (Relational Mapping)**: No LLM. Direct PostgreSQL → Neo4j MERGE with Schema Constraints (O(log N) B-Tree).
- **Part B (Semantic Extraction)**: **Gemini 2.5 Flash** (Cloud API) for zero-shot Knowledge Graph extraction from enriched descriptions. Semaphore(15) for cloud-speed throughput.
- **Part C (Vector Embeddings)**: **nomic-embed-text** (Local Ollama) for 768-dimension vector index generation. Stays local for data privacy.

**Safety Mechanisms:**

- N4eo4j Schema Constraints on `Product.id` and `Category.id` (prevents O(N²) full scan).
- `pg_id` (PostgreSQL integer) as the universal primary key across both databases (prevents title collision).
- `chunk_size = 500` with per-chunk Neo4j commit (enables safe Ctrl+C interruption).
- Checkpoint resumption: skips products that already have non-BELONGS_TO edges on restart.

### Make & CLI Automation

#### [MODIFY] `Makefile`

- `graphrag-db-up` / `graphrag-db-down`: Start/stop dual-database cluster.
- `graphrag-generate-data`: Run Node.js generator _AND_ execute Python Diversity Injection over resulting `.jsonl`.
- `graphrag-seed-postgres`: Load 100k generated JSONL into PostgreSQL natively.
- `graphrag-etl`: Run full async API LLM extraction & mapping loop (safety extraction limits attached to env `EXTRACT_LIMIT`).
- `graphrag-neo4j-clean`: Wipe Neo4j for fresh ETL.
- `graphrag-rebuild`: One-command full rebuild (clean Neo4j → ETL).

## Lessons Learned

> [!CAUTION]
> **"Data Quality First" Rule**
> Always validate the semantic content of your source data BEFORE building extraction pipelines. We spent hours optimizing a Ferrari engine only to discover the fuel tank was filled with water (`faker.lorem.paragraph()`). The correct order is:
>
> 1. Verify data has semantic value → 2. Build pipeline → 3. Scale up.

> [!WARNING]
> **Lab ≠ Load Test**
> 104,000 rows of 130 duplicated products only validates infrastructure throughput (memory, write speed, crash recovery). It does NOT validate knowledge graph diversity, search relevance, or RAG answer quality. True functional testing requires unique, semantically diverse product data.
